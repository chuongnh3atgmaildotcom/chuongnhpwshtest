 see result of ps://developers.cloudflare.com/api/resources/firewall/subresources/access_rules/methods/create/#(params)%20default%20%3E%20(param)%20zone_id%20%3E%20(schema)
 id, email, type (org/zone)
 Scope is the entity (container) the rule is attached to
 scenario 1: Two access rules in the same entity scope → same scope.id?
  Yes. If you create two access rules in the same scope (e.g. same account, both via accounts/{account_id}/firewall/access_rules/rules with the same account_id), both rules will have the same scope in the API response: same scope.id, same scope.type.
 scenario 2: if I create 2 kind-of-rules that have scope object as notion of scope, in same entity, they would have same scope.id?
  If both rule types use a scope object (like IP Access Rules do) and both are created in the same entity (same account or same zone), then in practice yes – they should have the same scope.id.
 scenario 3: if I create 1 access_rule in account and 1 access_rule in 1 of that accounts' zone, the first access rule is trickled down to zone, so they would have same scope.id?
  No. They have different scope.id (and different scope.type).
 The essence of my question is: the id itself is not identical, but uniqueness of using scope.id and entity.id in many scenarios is identical
  If the only goal were “uniquely identify the entity,” a single entity id could be enough. The reason Cloudflare still uses a separate scope.id is mostly system boundaries, not uniqueness:
  The firewall/access-rules service can keep its own identifier for “where this rule is attached” and store/index by that.
  It doesn’t have to depend on the Accounts (or Zones/User) API’s notion of “entity id,” or guarantee that the two are the same.
  So you get same uniqueness (one id per entity in each world) but different id spaces: firewall scope.id vs account/zone/user entity id. That avoids coupling and lets each system evolve its ids independently.
  So: the uniqueness of using scope.id vs entity id is often the same; the id itself differs because they live in different systems, not because the concept of “one scope per entity” is different.




Use Get-AzResource with the web app resource type so you only get lightweight resource records, then count them. That avoids loading full web app config (unlike Get-AzWebApp).
    Count web apps in the current subscription:
    (Get-AzResource -ResourceType Microsoft.Web/sites).Count
    Count without keeping the list in memory (stream and count):
    @(Get-AzResource -ResourceType Microsoft.Web/sites).Count
    Only slots (exclude app root):
    Slots are Microsoft.Web/sites/slots. To count only the main apps (no slots):
    (Get-AzResource -ResourceType Microsoft.Web/sites | Where-Object { $_.ResourceId -notmatch '/slots/' }).Count
    Get-AzResource -ResourceType Microsoft.Web/sites returns minimal resource objects (Id, Name, ResourceGroupName, Type, etc.), not full WebApp objects, so it’s more efficient than Get-AzWebApp when you only need a count. Azure doesn’t expose a “count only” API, so listing with this resource type is the usual approach.

1. cf entity taxonomy: resource, scope, permission, entity, services (produt), component(engine)
    in domain knowledge, entity is general term ~ a top resource. 
    top resource: zone + account: have idnetity and ownership
    sub resrouces: A resource defines: A configurable capability, a functional object
        acc: global
            - Administrative Entities: Billing, Account Settings, Audit Logs.
            - Global Security Entities: Account WAF, Account Firewall Access Rules, DNS Firewall.
            - Developer/Compute Entities: Workers Scripts, Workers KV Storage, D1, Workers AI.
            - Zero Trust Entities: Zero Trust, Access: Service Tokens, Cloudflare Tunnel.
        zone: specific web
            - Routing/Traffic Entities: DNS, Page Rules, Origin Rules, Single Redirect.
            - Domain Security Entities: Zone WAF, Page Shield, SSL and Certificates.
            - Domain Performance Entities: Cache Rules, Response Compression.

    A service is a commercial and functional classification used for dashboard organization and product grouping. 
    Selecting a service in the user interface causes the system to map that selection to a predefined array of resources. For example, the Zero Trust service maps directly to specific resources including Access Service Tokens and Access Audit Logs. During an API request, the authorization engine evaluates only the entity scope, the targeted resource, and the assigned permission
    vice-versa: You updated one Resource (Account Filter Lists) within one Entity (your Account). Because the WAF, Zero Trust, Rate Limiting, and Magic Transit "Services" are just operational layers that query the same centralized list, they all inherit the change simultaneously without needing independent updates.
